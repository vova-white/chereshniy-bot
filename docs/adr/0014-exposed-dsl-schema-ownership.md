# Exposed DSL schema ownership

The backend uses Exposed DSL without DAO. Table definitions live in the persistence package of the bounded context or subcontext that owns the data, while the app composition root orchestrates database setup in dependency order.

**Consequences**

- There is no global `Tables` package or shared table registry containing all domain tables.
- Each context owns its Exposed table objects, row mapping, and persistence adapters.
- Exposed `ResultRow` and persistence row DTOs do not cross persistence boundaries; repositories return domain/application models or explicit projection models.
- Each context exposes a persistence schema object that lists its owned tables; the app database module aggregates those table lists for dev/test schema setup.
- Context schemas may expose setup hooks, but the hooks must stay persistence-specific and must not contain domain workflow logic.
- Database table names use context/module prefixes to avoid collisions in the shared physical database.
- Cross-context foreign keys are allowed only for stable identity relationships and must not become a substitute for application interfaces.
- UUID primary keys are generated by application/domain code before insert rather than relying on database-generated IDs.
- The app database module may call context schema setup hooks, but domain modules must not depend on the app package.
- Exposed `SchemaUtils` is acceptable for dev/test bootstrap while the schema is young.
- Production deployments should use explicit reviewed migrations before real user data depends on the schema.
